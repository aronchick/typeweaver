1. What this project is

TypeWeaver is a typography robustness lab.

The core thesis is:
- most font tools optimize for ideal mockups
- TypeWeaver should optimize for how fonts survive reality

The intended long-range product is:
- ingest fonts
- evaluate them under hostile conditions
- explain where they fail
- compare candidates
- generate or repair stronger variants
- eventually export downloadable winning fonts

The current repo is still early-stage. The trustworthy wedge is deterministic evaluation, not unconstrained generation.

2. Product strategy

Do not mix "cool font generation" into the foundation too early.

The correct build order is:
- evaluation kernel first
- explainable scoring second
- deterministic hostile-condition matrix third
- friendly permutation explorer UI fourth
- comparison workflow fifth
- guided repair generation sixth
- open-ended generation and adversarial optimization later

The product should feel friendly and fun on the surface, but rigorous underneath.

UI principle:
- approachable, specimen-like, warm, visual

Engine principle:
- deterministic, explainable, reproducible

3. Scope discipline

Original Phase 1 constraints matter.

Phase 1 allowed:
- local font ingestion
- license normalization
- fixed Latin corpus rendering
- two benchmark profiles
- JSON report output

Phase 1 explicitly did not allow:
- font generation
- OCR scoring as a core feature
- URL auditing
- cloud infrastructure
- broad website product scope

Important repo reality:
- the repo already contains more than the original Phase 1 brief allowed, including API/UI and OCR-adjacent code
- do not let those extra surfaces distract from strengthening the evaluator

7. What to build next in code

The next implementation push should do these things:

A. Introduce a real scenario/scoring model
- preserve the current two profile names for compatibility
- internally evolve them toward a richer scenario abstraction
- do not hardcode more profile-specific bias logic

B. Upgrade report schema
- move beyond a single top-line score plus notes
- include structured subscores and failure diagnostics
- preserve deterministic JSON

C. Keep the UI fed by real evidence
- do not invent decorative metrics in the backend
- if a UI card exists, it should correspond to real report fields

D. Leave generation for later
- guided repair is acceptable after scoring and attack-matrix credibility exist
- open-ended font generation is not the next step

8. Architectural direction

The correct architecture is:

Layer 1: Registry and provenance
- ingest files
- normalize licenses
- classify approval status

Layer 2: Render and corpus
- fixed Latin corpus
- confusion pairs
- deterministic preview artifacts

Layer 3: Scenario engine
- conditions such as size, contrast, blur, raster pressure
- named scenario bundles built from reusable primitives

Layer 4: Scoring engine
- overall score composed from visible subscores
- condition-aware output
- explainable penalties

Layer 5: Product presentation
- UI and API should surface scenario, score breakdown, and failure explanation

Later layers:
- comparison frontier
- repair generation
- adversarial optimization
- exportable variants

9. UI guidance

The product should not feel like a compliance dashboard.

Desired qualities:
- warm and inviting
- specimen-lab energy
- easy to understand
- visually direct
- a little playful without becoming unserious

Do:
- use human-readable scenario names
- emphasize "show me where it breaks"
- surface worst-case behavior
- let the user compare environments

Do not:
- dump raw JSON as the primary experience
- present everything as admin tables
- treat the UI as a thin CLI wrapper forever

10. Scoring guidance

Scoring should be:
- interpretable
- decomposable
- target-aware
- useful for future optimization loops

Likely v1 scoring dimensions:
- overall_score
- robustness_score
- distinctiveness_score
- cost_score
- condition_breakdown
- worst_case_score
- top_failure_summary

Likely future additions:
- stability_score
- machine_readability_score
- intent_fit_score

Do not hide the mechanism behind a magic number.

11. Attack matrix guidance

The future matrix should be built from reusable deterministic primitives.

Core primitives under consideration:
- blur
- low contrast
- small size
- raster degradation
- compression artifacts
- glare or bloom
- motion blur
- camera capture
- print spread
- noise overlays

Scenario bundles should represent real environments, not raw toggles.

Examples:
- web_light_default
- mobile_dark_low_contrast
- later: browser/device/display/environment bundles

15. Non-negotiable implementation principles

- Keep deterministic behavior where possible.
- Prefer explicit data structures over speculative abstraction.
- Keep modules small and concrete.
- Prefer simple CLI-first/core-first design.
- Match UI claims to real engine output.
- Do not expand scope just because the repo already contains extra crates.
- Build trust in evaluation before pushing generation.
